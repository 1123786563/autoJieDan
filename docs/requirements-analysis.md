## 1. 项目概述与目标

### 1.1 项目背景

本项目旨在构建一个**自主生存型AI接单代理系统**，通过融合两个核心子系统实现端到端的软件开发服务自动化：

| 系统 | 语言 | 定位 | 核心职责 |
|------|------|------|----------|
| **Automaton** | TypeScript | 自主生存型AI经济主体 | 经济自主、商务决策、系统治理、区块链操作 |
| **Nanobot** | Python | 超轻量级AI代理框架 | 技术开发、代码生成、测试执行、客户沟通 |

### 1.2 核心目标

1. **经济自主**: 系统能够自主在外包平台接单、评估、签约并完成交付，实现经济上的自我维持
2. **端到端自动化**: 从需求获取到代码交付的完整自动化流程
3. **多平台支持**: 支持Upwork、Freelancer等多个外包平台
4. **安全协作**: 基于ANP协议的去中心化安全通信

### 1.3 目标任务类型

| 任务类型 | 匹配度 | 优先级 |
|----------|--------|--------|
| 标准化Web应用开发 | 95% | P0 |
| 自动化脚本与工具 | 92% | P1 |
| 智能合约与Web3应用 | 88% | P2 |

**BIZ-002 准确率定义**:
```
Accuracy = (TP + TN) / (TP + TN + FP + FN)

其中:
- TP (True Positive):  正确接受的好项目
- TN (True Negative):  正确拒绝的差项目
- FP (False Positive): 错误接受的差项目
- FN (False Negative): 错误拒绝的好项目

验证数据集: 50个人工标注的历史项目案例
```

```
Automaton: did:anp:automaton:main

```

#### 核心消息类型

| 消息类型 | 方向 | 用途 |
|----------|------|------|
| TaskCreate | Automaton → Nanobot | 创建新任务 (Genesis Prompt) |

| CapabilityQuery | 双向 | 能力查询 |
| BudgetUpdate | Automaton → Nanobot | 预算更新 |

#### Genesis Prompt消息结构

```json
{
  "@context": ["https://w3id.org/anp/v1"],
  "@type": "ANPMessage",
  "id": "01HXYZ123456789",
```

## 10. RALPLAN-DR摘要

### 10.1 原则 (Principles)

| # | 原则 | 说明 |
|---|------|------|
| P1 | **自主优先** | 系统设计以自主运营为核心，最小化人工干预 |
| P2 | **安全通信** | 所有跨系统通信必须加密、签名、可验证 |
| P3 | **优雅降级** | 资源不足时自动降级而非崩溃 |
| P4 | **增量交付** | 采用迭代交付模式，快速验证 |
| P5 | **职责分离** | Automaton管商务，Nanobot管技术，边界清晰 |

### 10.2 决策驱动因素 (Decision Drivers)

| # | 驱动因素 | 权重 | 影响决策 |
|---|----------|------|----------|
| D1 | **经济可持续性** | 高 | 生存状态管理、资源优化策略 |
| D2 | **交付质量保证** | 高 | 测试自动化、代码审查流程 |
| D3 | **通信安全性** | 高 | ANP协议选择、加密方案 |

### 10.3 可行方案分析

#### 方案A: ANP协议 + 双系统架构 (推荐)

**描述**: 采用ANP协议作为通信层，Automaton(TypeScript)负责商务，Nanobot(Python)负责技术

**优点**:
- 去中心化身份，无需中心服务
- 语义互操作性强，JSON-LD标准
- 原生端到端加密
- 可扩展至多Agent网络
- **优雅降级支持**:
  - 优先级P3的元协议层(ANP-007)已实现，可在需要时简化
  - 加密层可降级为传输层加密(TLS only)
  - 协议协商失败时可回退到默认协议版本
  - 多Agent网络(ANP-008)可延后实现，不影响核心功能

**缺点**:
- 实现复杂度较高
- 开发周期较长(6周，含1周缓冲)
- 需要维护两套代码库

**风险评估**: 中等风险，已有基础实现

---

#### 方案B: HTTP REST + WebSocket通信

**描述**: 使用传统HTTP REST API + WebSocket进行系统间通信

**优点**:
- 实现简单，开发周期短(2周)
- 调试方便，工具成熟
- 学习成本低

**缺点**:
- 缺乏标准化身份管理
- 需要额外实现加密层
- 扩展性受限
- 未来扩展多Agent需重构
- **具体受限场景**:
  - **多Agent广播**: 无原生支持，需自行实现消息路由和订阅机制
  - **跨域身份验证**: 无DID标准，需依赖传统OAuth/JWT，难以实现去中心化
  - **端到端加密**: 需额外实现密钥交换和消息加密，增加安全风险
  - **语义互操作**: 无JSON-LD支持，能力描述需自定义Schema
  - **协议协商**: 无标准化机制，版本兼容性难以保证

**风险评估**: 低风险但技术债务高

---

#### 方案C: MCP (Model Context Protocol)

**描述**: 使用MCP协议进行Agent间通信

**优点**:
- 专为LLM工具集成设计
- 社区支持良好

**缺点**:
- 主要面向工具集成，非Agent协作
- 缺乏经济/生存场景支持
- 无内置身份管理

**风险评估**: 不适合当前场景

---

### 10.4 方案选择建议

**推荐方案A (ANP协议)**

| 维度 | 评分 | 说明 |
|------|------|------|
| 安全性 | 9/10 | 原生加密、DID身份 |
| 可扩展性 | 9/10 | 支持多Agent网络 |
| 实现复杂度 | 6/10 | 较高但已有基础 |
| 开发周期 | 7/10 | 6周(含缓冲)，可接受 |
| 未来适配性 | 9/10 | 标准化协议 |
| 降级能力 | 8/10 | 多层优雅降级支持 |

**选择理由**:
1. 已有ANP基础实现(did.ts, encryption.ts, types.ts等)
2. 去中心化身份符合自主Agent理念
3. 端到端加密满足安全需求
4. 可扩展至未来多Agent协作网络
5. 支持多层优雅降级，降低风险

**替代方案触发条件**:
- 如果6周内无法完成核心功能，可回退到方案B
- 如果安全要求降低，可简化加密层
- CP1/CP2/CP3检查点失败时，评估降级或回退策略
